# harvest-lessons（教訓収穫スキル）

修正の歴史を資産に変えるフライホイール。git 履歴やレビュー指摘から「繰り返されているミス」「ルールがあれば防げたミス」を抽出し、モデル非依存の品質資産（hooks / rules / skills / docs）として定着させる。

## 引数

- 引数なし → カレントリポジトリの全履歴
- `--since <rev|date>` → 指定リビジョン/日付以降に限定（前回実行以降の差分収穫）
- `owner/repo` → GitHub 上の別リポジトリ（読み取り専用、`gh api` または一時ディレクトリへの shallow clone）。**github.com 上の SSO 組織もこの形式**（`gh auth status` で認証と SSO 認可を事前確認）
- `--host <ghe-host> owner/repo` → GitHub Enterprise 等の別ホストのみ（事前に `gh auth status --hostname <host>` で読み取り認証を確認する）

## 手順

### Step 1: 修正履歴の収集

1. `git log --oneline` で全体像を把握する
2. fix 系コミット（fix:, 修正, 解消, revert 等）を特定し、代表的なもの（10〜20件）の実差分を確認する
3. 同一ファイルが繰り返し修正されている「ホットスポット」を特定する

調査量が多い場合はサブエージェント（codebase-analyst）に委任する。

### Step 2: パターン分類

各修正を以下に分類する:

- **再発パターン**: 同種のミスが2回以上 → 昇格候補（最優先）
- **予防可能パターン**: 1回だが事前ルールで機械的に防げた → 昇格候補
- **ノイズ**: 一度きり・外部要因・設計判断の転換 → 除外（除外理由も記録する）

### Step 3: 裏取り（必須）

昇格候補のパターンごとに、**現行コードで事実確認**する。

- 「現在も該当コードが存在する」という指摘は grep / Read で実在を確認する
- 過去に修正済みのものは「再発防止」目的であることを明記する
- 検証できない・誤りと判明した候補は除外する

履歴分析エージェントの報告を鵜呑みにしない。誤った教訓の定着は無価値どころか有害。

### Step 4: 強制力の階段への振り分け

検証済みパターンを、上から順に「降ろせる最も強い層」へ振り分ける。

| 層 | 仕組み | 振り分け基準 |
|---|---|---|
| hooks / lint / CI | 決定的（モデル不要） | grep やコマンドで機械判定できる |
| .claude/rules/ | path 連動の自動ロード | 「このファイルを触るときは必ず」と言える |
| skills の観点 | タスク起動時にロード | タスク手順・レビュー観点に属する |
| docs / memory | 参照知識 | 上記に乗らない背景・経緯 |

rules には**過去の不具合事例**（コミット ID・現象・原因）を必ず添える。事例つきルールは弱いモデルほど効果が大きく、強いモデルにも害がない。

### Step 5: 成果物の提示と反映

1. 振り分け結果を「パターン / 根拠コミット / 提案する層 / 提案内容」の一覧でユーザーに提示する
2. 承認されたものを反映する（rules 新設・追記、hook スクリプト追加、skill 追記）
3. 複数リポジトリで通用する汎用資産は `/push-asset` での配信元昇格を提案する

## 外部・会社リポジトリのガバナンス（必須）

### 認証

- github.com 上の SSO 組織: 通常の `gh auth login` ＋組織への SAML SSO 認可で参照できる
- 読み取り専用を厳密に保証する場合は fine-grained PAT（contents: read-only）を `gh auth login --with-token` で使う
- アクセスは**読み取り操作**（log / diff / api の参照、shallow clone）に限定する。書き込み系の操作は行わない

### 教訓の抽象化

抽出した教訓は**汎用パターンに抽象化**して保存する。以下を資産に含めてはならない:

- コード断片のそのままの引用
- 社名・プロダクト名・内部システム名・人名
- 業務ドメイン固有の情報

抽象化の例: 「○○社の△△ API のリトライ漏れ修正」→「外部 API 呼び出しにはタイムアウトとリトライ方針を明示する」。判断に迷う場合は保存せず、ユーザーに確認する。

### サニタイズ検証（保存前の最終ステップ・必須）

外部リポジトリ由来の資産を保存・提示する前に、生成したテキスト全体を機械的に検証する:

1. **固有名チェック**: 組織名・プロダクト名・内部システム名・人名・社内 URL・リポジトリ名を grep で検索し 0 件であることを確認する（対象組織の具体的な語をユーザーに確認してチェックリスト化してよい）
2. **コード一致チェック**: 対象リポジトリのコードと行単位で一致する断片が無いことを確認する。識別子名の引用も避け、概念名に言い換える
3. **結果の明記**: 検証の実施と結果を報告に含める（例: 「サニタイズ検証済み: 固有名 0 件・コード一致 0 件」）

検証で 1 件でも検出された場合は保存せず、抽象化からやり直す。

## 注意事項

- 定期実行する場合は `--since` で前回以降に絞り、重複抽出を避ける
- 既存の rules / skills と重複する教訓は新設せず既存ファイルに追記する
- ルールの肥大化に注意。常時ロードされる層（rules）は「実際に再発しているもの」だけに絞る
