# PR 作成スキル

## 原則

- **PR作成はこの skill 経由が必須**（メイン/サブ問わず）。サブエージェントに実装を委譲する場合も、委譲プロンプトに『PR作成は pr skill を起動して行う』を明記するか、PR作成ステップをメイン側で行う。検問 hook（PreToolUse で `gh pr create` を検査する gate）を配線したリポジトリでは、直叩きの `gh pr create` は拒否される
- 独自の定型 PR フローを持つ skill は例外として、通行証プロトコル（下記手順 2〜3）に従った上で独自フローを続行してよい。**例外は現時点で push-asset のみ**（追加はこの skill の改訂で行う）。例外 skill では本文提示・`pr-review-loop` の要否を当該 skill のフロー文書が規定する（通行証プロトコルは免除不可）
- PR 本文はレビュアーと未来の自分のための一次資料。**何をしたかだけでなく「削った/残した・選んだ/捨てた判断の理由」を本文に残す**
- タイトルは 70 文字以内。ベースブランチが適切か（main 直か、開発ブランチか）を確認する
- 未コミットの変更があれば先にコミットするか確認する。生成した本文はユーザーに提示してから作成する
- 自律実行（ユーザーが対話にいない委譲・バッチ）では『本文をユーザーに提示してから作成』は事後報告に代えてよい。**テンプレ本文・`pr-review-loop` の起動は省略不可**
- **PR 作成後は `pr-review-loop` skill を自動起動**し、レビュー → 検証 → 修正 → 再レビューを回す（ユーザーが「レビュー不要」と明示した場合のみスキップ）。ループ収束後もマージは手動

## テンプレート

```markdown
## 概要
{変更の目的と概要を 1〜3 行で}

## 変更内容
{主な変更点を箇条書きで}

## 判断理由
{採った設計・削った選択肢とその理由}

## テスト
- [ ] {テスト項目}
```

## 手順（通行証プロトコル）

検問 hook を配線したリポジトリでは、この skill が立てる通行証マーカーだけが `gh pr create` を通す。hook が無いリポジトリでも同じ手順で作成する（マーカーは無害な一時ファイル）:

1. 本文・ベースブランチを確定し、未コミット変更があれば先にコミットする
2. **PR 作成の直前に** `touch .claude/pr-skill-active`（通行証マーカーを立てる。**独立した Bash 呼び出し**で実行）
3. `gh pr create` と通行証の片付けを**同一コマンド**で実行する（`;` 連結で片付けを成否によらず走らせ、`(exit $s)` で作成側の終了コードを呼び出しに返す）

```bash
touch .claude/pr-skill-active   # ← ①独立した Bash 呼び出しで先に実行
```

```bash
gh pr create --title "..." --body-file <file>; s=$?; rm -f .claude/pr-skill-active; (exit $s)   # ← ②作成と片付けを同一呼び出しで（終了コードは create のものを返す）
```

⚠ 運用上の注意（実運用で確認済みの落とし穴）:
- **hook は PreToolUse（実行前）評価**のため、`touch` を PR 作成と同一コマンドに `&&` で並べても間に合わずブロックされる。touch は**必ず独立した Bash 呼び出し**にする。逆に `rm` は PR 作成と同一コマンドに置いてよい（評価時点でマーカーは有効なので通る。コマンドが実行されれば成否問わず片付けが走る）。`rm -f` を素の末尾に置くと create の失敗が exit 0 に化けるため、必ず `s=$?` → `(exit $s)` の形で終了コードを伝える
- コマンド文字列やコミットメッセージに「gh pr create」という語句が含まれるだけでも hook が反応する（偽陽性）。コミットメッセージは `git commit -F <file>`、本文は `--body-file` でファイル渡しにする

マーカーの失効（mtime が 60 分超で無効扱い）は**検問 hook 側の実装**であり、hook が無いリポジトリではマーカーは自動では消えない — 手順 3 の片付けを必ず行う。マーカーは `.gitignore` に登録してコミットしないこと（未登録なら `.claude/pr-skill-active` を追記する。誤コミットすると checkout のたびに mtime が更新され、検問が常時許可に退行する）。

## チェックリスト

- [ ] `git log main..HEAD --oneline` と `git diff main...HEAD --stat` を本文生成の根拠にした
- [ ] 判断理由セクションが「自明」以外の決定を網羅している
- [ ] ドラフトにするかを確認した

## 過去事例

- 判断理由を本文に書いた PR は、後日の方針見直し（資産の整理・skill の取捨選択）で一次資料としてそのまま使えた。書かなかった PR は経緯の再調査から始まった
