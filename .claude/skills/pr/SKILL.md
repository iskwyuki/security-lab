# PR 作成スキル

## 原則

- PR 本文はレビュアーと未来の自分のための一次資料。**何をしたかだけでなく「削った/残した・選んだ/捨てた判断の理由」を本文に残す**
- タイトルは 70 文字以内。ベースブランチが適切か（main 直か、開発ブランチか）を確認する
- 未コミットの変更があれば先にコミットするか確認する。生成した本文はユーザーに提示してから作成する
- **PR 作成後は `pr-review-loop` skill を自動起動**し、レビュー → 検証 → 修正 → 再レビューを回す（ユーザーが「レビュー不要」と明示した場合のみスキップ）。ループ収束後もマージは手動

## テンプレート

```markdown
## 概要
{変更の目的と概要を 1〜3 行で}

## 変更内容
{主な変更点を箇条書きで}

## 判断理由
{採った設計・削った選択肢とその理由}

## テスト
- [ ] {テスト項目}
```

## チェックリスト

- [ ] `git log main..HEAD --oneline` と `git diff main...HEAD --stat` を本文生成の根拠にした
- [ ] 判断理由セクションが「自明」以外の決定を網羅している
- [ ] ドラフトにするかを確認した

## 過去事例

- 判断理由を本文に書いた PR は、後日の方針見直し（資産の整理・skill の取捨選択）で一次資料としてそのまま使えた。書かなかった PR は経緯の再調査から始まった
