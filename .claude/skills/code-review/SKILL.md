# コードレビュースキル（マルチエージェント版）

## 使い方

- `/code-review` — Full レベル（デフォルト、7エージェント並列）
- `/code-review standard` — Standard レベル（4エージェント）
- `/code-review lite` — Lite レベル（1エージェント）
- `/code-review HEAD~3` — コミット範囲指定（レベルは Full）
- `/code-review pr 123` — PR の差分をレビュー（`gh pr diff 123` で取得）
- `/code-review standard HEAD~3` — レベル + 範囲の組み合わせ

## レベル構成

| レベル | エージェント | 用途 |
|------|---------|------|
| **Lite** | code-reviewer | typo修正、軽微な変更 |
| **Standard** | code-reviewer + design-checker + requirements-checker + rules-checker | 通常の開発 |
| **Full** | 全7エージェント並列 | コミット前の本格レビュー（デフォルト） |

> **設計原則**: 各レビューエージェントは「全件報告（recall 重視）」、その後の検証パス（Step 5）が「裏取り（precision 回復）」を担う。**全件報告と検証パスは必ずセットで運用すること**。発見段階で黙って捨てられた真の指摘は下流で回収できないが、誤指摘は検証パスで棄却できる。

## ワークフロー

### Step 1: 引数パース

引数からレベルとdiff範囲を判定する:
- レベル指定がなければ `full`
- diff範囲指定がなければ作業ツリーの全変更（`git diff HEAD` + 未追跡ファイル）

### Step 2: 変更差分の収集

1. `git diff HEAD`（または指定範囲）で差分を取得。PR 指定時は `gh pr diff <番号>` で取得
2. `git status --porcelain` で未追跡ファイルを検出、`Read` で読み込む（PR 指定時はスキップ）
3. 以下は除外:
   - `pnpm-lock.yaml`, `package-lock.json` 等のロックファイル
   - `.png`, `.jpg`, `.ico` 等のバイナリファイル
   - 自動生成ファイル（マイグレーション、ビルド成果物等）

### Step 3: コンテキスト収集

各エージェントに渡す共通コンテキストを準備する:

1. **変更ファイルの import 先** — 型定義やユーティリティの参照先を読む
2. **プロジェクトルール** — `.claude/rules/` 配下のルールファイルを読む
3. **アーキテクチャ** — `docs/architecture.md` を読む（存在する場合）
4. **関連 Issue** — コミットメッセージやブランチ名から Issue 番号を推定し、`gh issue view` で要件を取得

### Step 4: エージェント並列起動

レベルに応じた Agent を **1つのメッセージで並列起動** する。
各エージェントには以下を渡す:
- 変更差分（git diff 出力）
- エージェント固有のコンテキスト（下記参照）
- レビュー観点の指示
- 出力フォーマットの指示（後述）

**重要**: エージェントは `subagent_type` を指定せず、`description` で役割を明示する。各エージェントにはリードオンリーで調査・分析のみを行うよう指示する（ファイル編集はしない）。

#### エージェント詳細

**1. code-reviewer（全レベル）**
```
役割: バグ、ロジックエラー、セキュリティ、パフォーマンス問題の検出
コンテキスト: diff + import先の型定義
観点:
- ロジックのバグや誤り
- セキュリティ脆弱性（XSS, インジェクション, 認証漏れ）
- パフォーマンス問題（N+1, 不要な再レンダリング, メモリリーク）
- エラーハンドリングの欠如
- 機密情報のハードコード
```

**2. design-checker（Standard以上）**
```
役割: アーキテクチャ適合性、既存パターンとの一貫性チェック
コンテキスト: diff + docs/architecture.md + .claude/rules/ + 既存の類似ファイル
観点:
- プロジェクトのアーキテクチャ・設計パターンに従っているか
- 既存コードと一貫性のある命名・構造になっているか
- 責務分離は適切か（ファイル・関数の粒度）
- .claude/rules/ のプロジェクト規約に違反していないか
- DRY: 既存のユーティリティや関数を再利用しているか
```

**3. requirements-checker（Standard以上）**
```
役割: Issue要件との突合、仕様漏れの検出
コンテキスト: diff + 関連Issueの本文（gh issue viewで取得）
観点:
- 変更がIssueの要件を満たしているか（機能漏れがないか）
- Issueに記載された制約・条件が実装に反映されているか
- Issueのスコープ外の変更が混入していないか
- 副作用や既存機能への影響はないか
※関連Issueが特定できない場合はスキップ
```

**4. silent-failure-hunter（Fullのみ）**
```
役割: サイレント障害の検出、エッジケース分析
コンテキスト: diff + 変更ファイル全文
観点:
- try-catch でエラーを握りつぶしていないか
- null/undefined チェックの漏れ
- 非同期処理の await 漏れ
- 境界値・空配列・空文字のハンドリング
- レースコンディションの可能性
- DB操作の失敗時にデータ不整合が起きないか
```

**5. type-analyzer（Fullのみ）**
```
役割: TypeScript型安全性の検証
コンテキスト: diff + 関連する型定義ファイル
観点:
- any 型の使用（回避すべき箇所）
- as キャストの妥当性（安全でないキャスト）
- union 型の網羅性（switch/if での未処理パターン）
- 型定義とランタイム値の乖離
```

**6. simplifier（Fullのみ）**
```
役割: 不要な複雑さの検出、簡素化の提案
コンテキスト: diff + 変更ファイル全文
観点:
- YAGNI: 将来のための過剰な抽象化
- 使われていないコード、到達不能なコード
- 単純化できるロジック（三項演算子のネスト、過剰なガード節）
- 不要なラッパー関数や中間変数
- 同じことを別の方法で2回書いていないか
```

**7. rules-checker（Standard以上）**
```
役割: プロジェクト規約（.claude/rules/）との機械的照合
コンテキスト: diff + .claude/rules/ 配下の全ルールファイル
観点:
- 各ルールファイルのチェック項目・禁止事項と diff を1項目ずつ照合する
- ルールに「過去の不具合」事例が記載されている場合、同型のコードが再導入されていないか確認する
- 違反は該当ルールファイル名を添えて報告する
※ .claude/rules/ が存在しない、または該当 path のルールがない場合はスキップ
```

#### エージェントへの共通出力指示

各エージェントのプロンプト末尾に以下を付与する:

```
## 報告方針（最重要）
あなたの役割は網羅（カバレッジ）であり、フィルタリングではない。確信が持てない指摘・軽微に見える指摘も省略せず報告すること。誤指摘は下流の検証パスが棄却するが、あなたが黙って落とした指摘は誰にも回収できない。各指摘に confidence（high / medium / low）を付与すること。

## 出力フォーマット
以下の形式で結果を報告してください。該当なしのセクションは省略してください。

### Critical（必ず修正すべき）
- `file:line` — 指摘内容（confidence: high|medium|low）
  → 修正案: ...

### Warning（推奨）
- `file:line` — 指摘内容（confidence: high|medium|low）
  → 修正案: ...

### Info（参考）
- `file:line` — 指摘内容

### Good（良い点）
- 良い設計判断・実装のピックアップ

指摘がない場合は「指摘事項なし」とだけ報告してください。
```

### Step 5: 検証パス（Critical / Warning の裏取り）

統合の前に、各エージェントから上がった Critical / Warning を検証する。**全件報告方針と検証パスは必ずセットで運用する**（検証パスを省略して全件報告だけ行うと誤指摘がそのままユーザーに届く）。

1. Critical / Warning の指摘ごとに検証エージェントを並列起動する（新しいコンテキスト、リードオンリー）。指摘が多い場合は1エージェントに複数指摘を割り当ててよい（指摘ごとに独立判断するよう指示する）
2. 各検証エージェントへの指示:
   ```
   以下のレビュー指摘を「反証」せよ。擁護ではなく反証を試みること。
   実際のコード・型定義・呼び出し元を Read / Grep で確認し、指摘が現行コードに当てはまるかを判定し、verdict を返せ:
   - confirmed: 指摘は正しい（根拠の行番号を添える）
   - refuted: 誤指摘（反証の根拠を必ず添える）
   - uncertain: 判断不能（何が分かれば判断できるかを添える）
   ```
3. verdict の処理:
   - **refuted** → 統合レポートから除外し、件数のみ「検証で棄却: N件」と記載
   - **uncertain** → 残すが Warning 以下に降格
   - Info / Good は検証パスの対象外

### Step 6: 結果の統合

全エージェントの結果を受け取り、以下の統合レポートを生成する:

```markdown
## コードレビュー結果（{レベル}）

対象: {ファイル数}ファイル（{追加行数}行追加 / {削除行数}行削除）
エージェント: {起動したエージェント名のリスト}

### Critical（必ず修正）
- **[code-reviewer]** `file:line` — 指摘内容
  → 修正案: ...
- **[silent-failure-hunter]** `file:line` — 指摘内容
  → 修正案: ...

### Warning（推奨）
...

### Info（参考）
...

### Good
...

### サマリー
Critical: N件 / Warning: N件 / Info: N件 / 検証で棄却: N件
```

**統合ルール:**
- 各指摘の先頭にエージェント名をタグとして付与し、confidence と検証 verdict を保持する
- 複数エージェントが同じ箇所を指摘した場合は1つにマージし、検出元を併記
- Critical > Warning > Info の順でソート
- 同一重要度内ではファイルパス順でソート
- Good は全エージェントの Good をまとめて記載

### Step 7: Critical の自動修正提案

検証済み（confirmed）の Critical が 1件以上ある場合、`AskUserQuestion` で確認する:

> Critical が N 件あります。自動修正しますか？
> - **修正する** → Critical を修正し、修正後に **再レビュー（Lite レベル）** を実行
> - **このまま** → レポート表示のみで終了

**修正する場合:**
1. 検証済み Critical の各項目を Edit ツールで修正
2. 修正完了後、Lite レベルで再レビュー（無限ループ防止のため Lite 固定）
3. Critical が 0 件になったらレビュー完了

### Step 8: 効果ログの記録（Track B・best-effort）

統合レポート（Step 6）の件数を用いて効果ログを 1 行追記する。**JSON は `log-effect.sh` が組む。手で JSON を書かない**（件数を引数で渡すだけ）。plugin ルートが解決できない環境ではスキップしてよい（決定的な主計測は pre-commit ゲートが担うため）:

```sh
LE="${CLAUDE_PLUGIN_ROOT:-}/hooks/log-effect.sh"
if [ -f "$LE" ]; then
  sh "$LE" --tool code-review --model "<現在の実行モデル ID>" \
    --critical <Critical件数> --warning <Warning件数> --info <Info件数> \
    --confirmed <検証 confirmed 件数> --refuted <検証で棄却した件数> \
    --diff-lines <差分の追加+削除行数> --repo-path "$(git rev-parse --show-toplevel 2>/dev/null || pwd)"
fi
```

生ログは各リポジトリの `.claude/state/harness-telemetry/`（非コミット）に貯まる。スキーマは [`docs/effect-log/SCHEMA.md`](../../../docs/effect-log/SCHEMA.md) を参照。

## 注意事項

- レビュー対象が20ファイル超の場合は、主要な変更に絞り、その旨を報告する
- 自動生成コードはレビュー対象外
- `.claude/rules/` のプロジェクト規約違反は Warning 以上で報告する
- 修正は Critical のみ。Warning / Info は報告のみで自動修正しない
- 関連 Issue が特定できない場合、requirements-checker はスキップする
- レビュー結果は日本語で出力する
