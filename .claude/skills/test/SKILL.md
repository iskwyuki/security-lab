# テスト実行スキル

## 引数

- 引数なし → 全テスト、ファイルパス → 該当のみ、`--failed` → 前回失敗分のみ

## 原則

- プロジェクトのテストランナーは設定ファイルから検出する（package.json / pyproject.toml / Cargo.toml / go.mod 等）。検出に迷ったらユーザーに確認する
- 実行に時間がかかる場合は、変更に関連するテストのみの実行を先に提案する
- 失敗時は「合計/成功/失敗/スキップ」のサマリーに続けて、失敗テストごとに**ファイル・エラー内容・推定原因・修正案**を報告し、修正するか確認する
- 環境セットアップが必要な場合は先に確認する

## チェックリスト（失敗解析時）

- [ ] 失敗が「コードのバグ」「テストのバグ」「環境差異（環境変数・DB・CI との差）」のどれかを切り分けた
- [ ] flaky の疑いがある場合、同一テストの再実行で再現性を確認した

## 過去事例

- **TZ 依存の偶然成功**: 日付境界ロジックのテストがマシンのタイムゾーンに依存し、開発機では通るのに本番（UTC コンテナ）で 9 時間ずれるバグを素通しした。日付境界を扱うテストは **TZ を明示固定**して検証する
- **時刻衝突による flaky**: 同一タイムスタンプのレコード並び順に依存したページングテストが間欠失敗した。時刻を使うフィクスチャは**決定的にずらして**生成する
